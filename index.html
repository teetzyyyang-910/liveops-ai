選擇遊戲類型與活動類型：決定 AI 使用的 benchmark 基準數據與失衡判斷邏輯。選擇不建議的活動組合時，欄位下方會出現橙色提示。
2
設定玩家層分佈：鯨魚 / 中度 / 輕度合計需等於 100%，影響獎勵層級設計方向。累積消費 / 累積儲值活動可額外設定 Tier 數量（3 / 5 / 7 / 10 層）。
3
（建議填）輸入歷史留存數據：有真實 D1 / D7 / D30 數據時，留存預測與 LTV 計算更準確。
4
點擊「生成分析報告」：獲得獎勵結構、失衡警告（over_rewarded / over_barriered / under_rewarded）、留存預測三份結果。

🚀 發行評估模組

1
填入基本設定（必填）：遊戲類型、目標市場、行銷預算（M1-M36 總計）、CPI。這四個欄位決定安裝規模基礎。
2
填入財務條件（建議）：License Fee、MGF、月 ARPU。月 ARPU 是最影響 3 年收入預測準確度的欄位，建議填入真實數據或行業 benchmark。
2
（選填）匯入逐月數據 CSV：表單頂部展開「匯入逐月數據」，上傳或貼上開發商提供的財務數據。
・支援欄位：month, nru, monthly_arppu, rr_of_mau, paying_rate
・點「📋 範本下載」取得格式，填入後「✓ 套用匯入」
・匯入後自動覆蓋 NRU 錨點和 ARPU 錨點欄位
3
選擇 ARPU 成長假設：月 ARPU 輸入欄下方選擇成長模式。
・固定不變（預設）：保守估算，整個 3 年月 ARPU 不成長
・緩慢/穩定成長：對數或線性曲線，反映付費玩家成熟化
・自訂錨點：填入 M1/M3/M12 真實月 ARPU，系統插值
選後即時預覽 M6/M12/M36 走勢。這個選擇對 3 年總收入影響最大。
4
填入 DAU/MAU 與月 MAU 留存率（核心！）：月 MAU 留存率（首月後）決定 MAU 衰退速度，是 3 年收入預測最關鍵的輸入之一。M1→M2 的留存率越高，整體收入越高。系統自動依遊戲類型預填 benchmark，可依真實數據覆蓋。
5
填入玩家留存（建議）：核心玩家 D1/D7/D30。非核心留空則自動套用核心×70%。用於 Cohort 分析與 LTV 里程碑計算。
6
（選填）進階設定 — 行銷花費曲線很重要：
・行銷花費曲線：決定每月預算分配，影響 M1-M6 MAU 高峰和停投後的衰退。前6個月通常 ROI 最佳。
・NRU 錨點模式：勾選後可手動填入 M1/M3/M6/M12 的 NRU 數字（開發商提供的計劃數），比用「預算÷CPI」反推更準確。
・M0 預熱期：費用佔比自動換算安裝數，計入初始成本
・AI 會自動比較不同投放策略的 ROI，並給出具體建議
7
點擊「生成發行評估報告」：系統先跑趨勢走勢模型（逐月精細計算），再將結果送 AI 評估。AI 會比較不同投放曲線的效果並給出具體建議。
8
查看五個 Tab 的結果：
・發行評估：GO/MAYBE/NO-GO + 趨勢模型 ROI + 投放策略建議
・預算規劃：三情境快速估算（方向性參考）
・風險分析：基於趨勢模型數字的具體風險 + 談判建議
・趨勢走勢 ✅：最重要！月度 MAU/收入/CP Margin/累積現金流，◉ 標記停投月
・Cohort 分析：D180 LTV 里程碑 + ROAS 三情境
💡 最重要的三個輸入
1. 月 ARPU 成長假設：固定 vs 對數成長，可影響 3 年收入 30-60%
2. 月 MAU 留存率（M1→M2）：決定停投後的 MAU 衰退速度
3. 行銷花費曲線：前6個月集中投放通常優於全年平均（AI 會自動比較）

・發行評估的 GO/NO-GO 和 ROI 基於「趨勢走勢」模型，不是快速估算
・公式邏輯請看「公式說明」Tab，benchmark 數字請看「Benchmark」Tab
⚠️ 重要提醒：所有未填入歷史數據的欄位均使用行業 benchmark 推算。數字適合方向性決策，不建議直接用於投資報告或合約談判依據。D40+ 數據為模型推算，非實測。

LiveOps 活動模組

Net Retention Impact (NRI)
EB = 0.2×時長分 + 0.3×獎勵吸引力 + 0.5×可參與性 FC = 0.4×難度指數 + 0.6×肝度強度 NRI = EB − FC
EB（參與提升）越高、FC（疲勞係數）越低，NRI 越高，對留存越正面。
留存預測
D7 新留存 = D7基準 × (1 + NRI) D30 新留存 = D30基準 × (1 + NRI × 0.6)
D30 的影響係數為 0.6，因為活動效果在長期會衰減。
LTV（LiveOps 模組）
LTV = ARPU × (1 + D1/100 + D7/100×1.5 + D30/100×3)

發行評估模組

安裝與用戶轉換
預估安裝數 = 行銷預算 / CPI NRU = 安裝數 × 轉換率（預設 75%）
轉換率反映安裝→啟動→創角的整體漏斗損耗，實際範圍約 60-85%。
月活躍用戶（MAU）— 最新公式
MAU(m) = Σ cohort(i).NRU × cohortMauFraction(m−i) cohortMauFraction(age): age=0: 1.0 // 新進用戶，第一個月全量 MAU age=1: RR_MAU // 用戶填入（例：MMO=40%） age≥2: RR_MAU × 0.58^(age-1) // 逐月衰減，0.58 為 VNG 驗證值
RR_MAU（月 MAU 留存率）= M1→M2 仍活躍的 MAU 比例。這是最影響 3 年收入的單一參數。Benchmark 見「Benchmark」Tab。M2+ 穩定留存 rrStable 依遊戲類型自動設定，不會因為 CS2 案例而固定化。
CP Margin（貢獻利潤率）
CP Margin = (淨收入 − 當月行銷費用 − License 月攤銷) / 毛收入 淨收入 = 毛收入 × (1 − Channel − Royalty − Server − Staff − Platform) License 月攤銷 = License Fee / 36（均攤 3 年） 判斷標準： ≥40% 優秀（高利潤，代理條件佳） ≥20% 合格（有貢獻利潤） 0~20% 有貢獻但偏低 <0% 該月賠錢運營（直接成本 > 毛收入） 注意：單月 CP > 0 ≠ 整體回本（未扣除 MGF 等固定成本攤銷）
對齊 VNG 發行財務定義：Contributing Profit = Net Revenue − Marketing − License Amortization
ARPPU 成長模型
// 對數成長（預設） ARPPU(m) = 基準值 × min(1.8, 1 + 0.8×ln(m)/ln(18)) // 線性成長 ARPPU(m) = 基準值 × min(2.0, 1 + (m−1)×0.02) // 分段成長 M1-6: 基準值 × (1 + (m−1)×0.06) M7-18: 上期末值 × (1 + (m−6)×0.015) M18+: 固定 1.48x 基準值
留存用戶越核心，付費意願越高。付費率同步成長：payRate(m) = 基準 × min(2.5, 1 + (m−1)×0.012)
MAU 修正：月底存活 vs 真實 MAU
// 問題：cohortSurvival() 返回「月底仍存活」比例 // 這等於 D30 留存概念，但 MAU 定義是「本月任何一天有登入」 // 兩者差距 = 1 / DAU/MAU 比例 // 修正公式 月底存活人數 = Σ cohort(i).NRU × cohortSurvival(age) 真實 MAU = 月底存活人數 / DAU/MAU 比例 // 範例：RPG，DAU/MAU=32% 月底存活 = 1000 人 真實 MAU = 1000 / 0.32 = 3,125 人 ← 差 3.1 倍 收入差距 = 3.1 倍（若用月底存活計算收入會嚴重低估）
⚠ 這是系統收入預測準確度的關鍵。請確保填入正確的 DAU/MAU 比例，系統會自動套用修正。
月毛收入 & 淨利
// 方法 A：已填月 ARPU（推薦，最準確） 月毛收入 = 真實MAU × 月ARPU(m) // 真實MAU = 月底存活 / DAU/MAU比例 // 方法 B：由 MPU × ARPPU 計算 MPU = 真實MAU × 付費率(m) 月毛收入 = MPU × ARPPU(m) // 淨利 發行商淨利 = 毛收入 − Channel − Royalty − Server − Staff − Platform − 行銷 − License − MGF
MAU 修正後，收入預測比舊版提高 2~4 倍（取決於 DAU/MAU 比例）。若發現收入數字大幅上升，這是正確的行為。
baselineLTV 計算方式（發行評估）
// 正確做法：dailyRetention 積分（與 Cohort 分析一致） ARPDAU = 月ARPU / (DAU/MAU × 30) LTV90 = Σ(day=0→90) dailyRetention(d1,d7,d30,day)/100 × ARPDAU // 3年總收入估算 3yr Gross = 安裝數 × LTV90 × ltvScaleFactor // ltvScaleFactor 依遊戲類型（D90後長尾係數）： // Casual=0.8 RPG=1.8 放置=2.0 SLG=3.5 // MMO=3.2 開放世界=2.0 射擊=1.4 MOBA=1.6 Party=0.9
ltvScaleFactor 反映 D90 後的長尾收入，高留存遊戲（SLG/MMO）係數更高。舊版使用固定係數 2.5 對所有類型，現已依遊戲類型分別設定。
自然量衰退（付費停投後）
organic(month) = peak_organic × (1 − decayRate)^monthsAfterStop
衰退率可選：3%（強品牌）/ 5%（一般）/ 8%（中等）/ 12%（快速）/ 20%（短期熱潮）
核心 vs 非核心玩家分群
// NRU 分群 核心玩家 NRU = M1 前三天安裝量 × NRU轉換率 // M1 前三天安裝量由月內安裝分布曲線決定 // 例：首週集中型 → 前三天佔 50%×3/7 ≈ 21.4% 非核心玩家 NRU = (M1 第四天起 + M2 + M3 安裝量) × NRU轉換率 // 留存預設值（若未填入非核心數據） ncD1 = coreD1 × 70% ncD7 = coreD7 × 70% ncD30 = coreD30 × 70% // 反映非核心用戶黏性通常低於核心用戶 30% // 累積 LTV（兩組各自獨立計算） LTV_core(N) = Σ(day=0→N) ret_core(day)/100 × ARPDAU LTV_nc(N) = Σ(day=0→N) ret_nc(day)/100 × ARPDAU // D90 / D180：D30 後指數衰減推算（終身價值估算用） decay = (D30/D7)^(1/23) retention(day) = D30 × decay^(day−30)
核心玩家代表首發爆量的高品質用戶，非核心代表後續買量用戶。兩組 LTV 差距越大，代表首發的用戶獲取效率越高，值得在首發集中投入更多預算。
ARPDAU 正確計算方式
// 錯誤做法（過度簡化） ARPDAU = 月ARPU / 30 ← 假設每天 DAU = MAU，不正確 // 正確做法 ARPDAU = 月ARPU / (DAU/MAU比例 × 30) // 範例：RPG 月ARPU=$5，DAU/MAU=32% ARPDAU = $5 / (0.32 × 30) = $5 / 9.6 = $0.52/天 // 若錯誤用 /30：$5/30 = $0.17/天 ← 低估 3 倍 // 方法 A：自填月 ARPU + DAU/MAU（推薦） ARPDAU = 月ARPU_用戶輸入 / (DAU/MAU × 30) // 方法 B：由 ARPPU × 付費率推算 月ARPU_估算 = ARPPU × 付費率 ARPDAU = ARPPU × 付費率 / (DAU/MAU × 30)
⚠ 不同遊戲的月 ARPU 差異極大（$0.1~20+）。DAU/MAU 比例決定了 ARPDAU 的分母，對 LTV 影響極大。優先使用真實後台數據。
Cohort LTV 插值與累積計算
// D1~D7：對數插值 t = ln(day) / ln(7) retention(day) = D1 + (D7−D1) × t // D7~D30：對數插值 t = (ln(day)−ln(7)) / (ln(30)−ln(7)) retention(day) = D7 + (D30−D7) × t // D30 之後：指數衰減 decay = (D30/D7)^(1/23) retention(day) = D30 × decay^(day−30) // 累積 LTV 到第 N 天（含安裝當天 Day 0） LTV(N) = Σ(day=0→N) retention(day)/100 × ARPDAU // Day 0：100% 用戶在線（安裝當天） // Day 1：D1留存% 用戶回來 // Day 7：D7留存% 用戶回來... // 範例（RPG，月ARPU=$20，DAU/MAU=32%） ARPDAU = $20 / (0.32×30) = $0.52 LTV D1 ≈ (1.0 + 0.45) × $0.52 ≈ $0.76 LTV D30 ≈ 累積留存面積 × $0.52 ≈ $5.4 ✓ // 若用月ARPU=$1.5（benchmark） ARPDAU = $1.5 / (0.32×30) = $0.039 LTV D30 ≈ $0.4 ← 差 13 倍
LTV 準確度由兩個變數決定：月 ARPU 和 DAU/MAU 比例。兩者都用真實後台數字，LTV 才有意義。

以下數據綜合自 AppsFlyer、Adjust、Sensor Tower、data.ai 年度報告及業界公開研究，取中位數區間。 僅作方向性參考，不代表任何特定遊戲的實際表現。

⚠ 關於月 ARPU 的重要說明
月 ARPU（每活躍用戶月均收入）= ARPPU × 付費率。但不同遊戲差異極大： 下表的 benchmark ARPU 是市場中位數，實際頭部遊戲可高出 10～20 倍。 例如某款 RPG 的真實 LTV30 ≈ $5.44，反推月 ARPU ≈ $20，而 benchmark 中位數僅 $1.5。
→ 發行評估模組中請務必填入自己的「月 ARPU」欄位，LTV 預測才有意義。

D90 / D180 LTV 估算說明

⚠ D40+ 為模型推算，非實測數據
D30 後的留存使用指數衰減公式推算，隨時間衰減速度由 D7→D30 的斜率決定。
D90 和 D180 的數字代表終身價值趨近值：
— 對高黏性遊戲（MMO / SLG）：D180 LTV 仍在增長，代表長期活躍用戶的真實價值
— 對低黏性遊戲（Casual / RPG）：D90 後數字趨近平緩，代表該類型遊戲的 LTV 上限
建議搭配真實後台的 D90 數據驗證，若真實值高於推算值，代表你的遊戲黏性優於業界均值。
NRU
New Registered Users，新註冊用戶數 = 安裝數 × 轉換率
MAU
Monthly Active Users，月活躍用戶，所有歷史 cohort 的存活用戶加總
MPU
Monthly Paying Users，月付費用戶數 = MAU × 付費轉化率
ARPU
Average Revenue Per User，每用戶平均收入（含免費用戶）
ARPPU
Average Revenue Per Paying User，每付費用戶平均消費（月），通常遠高於 ARPU
LTV
Lifetime Value，用戶全生命週期價值，衡量每個安裝帶來的長期累積收益
CPI
Cost Per Install，每次安裝成本，行銷效率的核心指標
pCPI
Paid CPI，付費渠道的安裝成本，排除自然流量後的真實投放成本
D1/D7/D30
第 1 / 7 / 30 天留存率，衡量用戶在安裝後特定天數仍活躍的比例
NRI
Net Retention Impact，活動對留存的淨影響值 = EB（參與提升）− FC（疲勞係數）
Cohort
同一時期獲取的一批用戶，追蹤這批用戶隨時間的行為變化
MGF
Minimum Guarantee Fee，最低保證金，代理合約中開發商要求的最低收入保障
License Fee
授權費，代理合約的前期固定費用，不依銷售額變動
Royalty
版稅 / 分潤，通常為毛收入的固定比例（本工具預設 23%）
Channel Fee
渠道費，應用商店（App Store / Google Play 等）抽取的比例
ROI
Return on Investment，投資回報率 = (淨利 − 投入) / 投入 × 100%
